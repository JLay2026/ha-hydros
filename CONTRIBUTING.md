# Contributing to JLay2026/ha-hydros

This is a hardening fork of [`Bitf1ip/ha-hydros`](https://github.com/Bitf1ip/ha-hydros). It exists to support the [Workshop dashboard project](https://github.com/JLay2026/HA-dashboard) and to feed hardening contributions back upstream.

The fork is set up so that:

- Fork's `main` is kept in sync with `Bitf1ip/ha-hydros:main` daily (auto fast-forward when no conflict; sync PR opened on divergence)
- Code-level hardening that belongs upstream is auto-PR'd to `Bitf1ip/ha-hydros` when the fork PR merges
- Fork-only changes (docs specific to our deployment, references to the Workshop dashboard) stay local

This file documents the conventions that make that flow work.

---

## Branch naming convention

The automation routes work based on the branch prefix.

| Prefix | Meaning | Auto-PR upstream? |
|---|---|---|
| `upstream/<topic>` | Code/behavior change that should also land in `Bitf1ip/ha-hydros` | **Yes** (on merge to fork's main) |
| `fork-only/<topic>` | Documentation or convention specific to this fork's downstream use | No |
| `sync/<date>` | Auto-generated by the daily upstream-sync workflow | No (manual review) |

**Examples:**

- `upstream/v0.4.0-stale-retention` → implements [#3](../../issues/3) (cloud-outage resilience); merging opens an upstream PR
- `fork-only/v0.4.1-readme-alert-discipline` → implements [#7](../../issues/7) (alert-template docs); stays on fork

Pick the prefix at branch creation. If you mix scopes in one branch, split them — the upstream PR will carry the entire branch contents, so a `upstream/*` branch must not contain fork-only changes.

## Upstream PR authorship rule

**Upstream PRs to `Bitf1ip/ha-hydros` are authored by JLay2026 personally, never by `github-actions[bot]` or `digispark-bot`.**

Why: contributor identity matters. Upstream maintainers track contributor history. The author of [v0.3.3's hub-methods fix](https://github.com/Bitf1ip/ha-hydros/blob/main/CHANGELOG.md) is credited as @JLay2026. Service-account PRs dilute that identity and weaken the contributor relationship.

### How the automation enforces this

`.github/workflows/upstream-pr.yml` opens cross-repo PRs using a fine-grained Personal Access Token stored as the `UPSTREAM_PR_TOKEN` repo secret. Because GitHub bases PR authorship on the API token, the PR is created as the PAT's owner (JLay2026) — not as `github-actions[bot]`.

The default `GITHUB_TOKEN` is **not used** for upstream PRs (only for fork-internal operations like the sync workflow).

### One-time PAT setup

1. Visit https://github.com/settings/personal-access-tokens/new (signed in as @JLay2026)
2. Token name: `ha-hydros-upstream-pr` (or similar)
3. Expiration: 90 days (rotate on schedule)
4. Repository access: **Only select repositories** → `Bitf1ip/ha-hydros`
5. Repository permissions:
   - **Pull requests:** Read and write
   - **Contents:** Read-only
   - **Metadata:** Read-only (auto-required)
6. Generate token → copy value
7. Add to fork as repo secret:
   - https://github.com/JLay2026/ha-hydros/settings/secrets/actions
   - Name: `UPSTREAM_PR_TOKEN`
   - Value: paste the PAT
8. Note the rotation date; renew before expiry

### Verification

After the first `upstream/*` branch merges and the workflow runs:

- Open the resulting PR on `Bitf1ip/ha-hydros`
- Confirm the **Author** field shows `JLay2026`, not `github-actions[bot]` or `digispark-bot`
- If it shows the wrong author, the PAT secret is missing or misconfigured — do not rely on the workflow until fixed

## Daily upstream sync

`.github/workflows/upstream-sync.yml` runs daily at 06:00 UTC (and on-demand via Actions tab → Run workflow).

Behavior:

- **No divergence:** fast-forwards fork's `main` to upstream `main`. No PR, no human review needed.
- **Conflict detected:** opens a `sync/<YYYY-MM-DD>` PR for manual review. Resolve conflicts on that branch, merge to bring fork `main` back in sync.
- **Fork commits ahead:** fork commits are preserved; sync only adds upstream commits. If a fork commit is later merged upstream, the next sync sees it as already-present and skips.

If the daily cadence is too noisy or too sparse, edit the cron expression in the workflow.

## Workflow for hardening items

For each issue in [the M2 tracker (#2)](../../issues/2):

1. **Branch:** `git checkout -b upstream/v0.4.0-<short-name>` (or `fork-only/...`)
2. **Implement:** code + tests + docs against the acceptance criteria in the issue
3. **PR to fork main:** open from the branch; reference the issue number; request your own review or self-merge after CI passes
4. **Merge to fork main:** `Squash and merge` keeps history clean
5. **Auto-action:**
   - For `upstream/*`: workflow opens the upstream PR. Watch the Actions tab; visit the new upstream PR to confirm author = JLay2026 and the title/body look right
   - For `fork-only/*`: nothing extra; merge is the end of the workflow
6. **Upstream PR review:** track the upstream PR. Comment, iterate, and merge per Bitf1ip's review cadence
7. **Upstream merge:** next daily sync brings the merge commit back into fork `main` cleanly (no-op since the commit is already present)
