# Contributing to JLay2026/ha-hydros

This is a hardening fork of [`Bitf1ip/ha-hydros`](https://github.com/Bitf1ip/ha-hydros). It exists to support the [Workshop dashboard project](https://github.com/JLay2026/HA-dashboard) and to feed hardening contributions back upstream.

The fork is set up so that:

- Fork's `main` is kept in sync with `Bitf1ip/ha-hydros:main` daily (auto fast-forward when no conflict; sync PR opened on divergence)
- Code-level hardening that belongs upstream is opened MANUALLY as a PR on `Bitf1ip/ha-hydros` after the fork PR merges
- Fork-only changes (docs specific to our deployment, references to the Workshop dashboard) stay local

This file documents the conventions that make that flow work.

---

## Branch naming convention

The branch prefix signals **intent**, not automation — see "Manual upstream PR submission" below for the actual flow.

| Prefix | Meaning | Upstream-bound? |
|---|---|---|
| `upstream/<topic>` | Code/behavior change that should also land in `Bitf1ip/ha-hydros` | **Yes** — open upstream PR manually after merge |
| `fork-only/<topic>` | Documentation or convention specific to this fork's downstream use | No |
| `sync/<date>` | Auto-generated by the daily upstream-sync workflow | No (manual review) |

**Examples:**

- `upstream/v0.4.0-stale-retention` → implements [#3](../../issues/3) (cloud-outage resilience); merge to fork main, then open upstream PR manually
- `fork-only/v0.4.1-readme-alert-discipline` → implements [#7](../../issues/7) (alert-template docs); stays on fork

Pick the prefix at branch creation. If you mix scopes in one branch, split them — the manual upstream PR will carry the entire branch contents (minus fork divergence the maintainer cherry-picks around), so an `upstream/*` branch must not contain fork-only changes.

## Manual upstream PR submission

**Upstream PRs to `Bitf1ip/ha-hydros` are opened manually by JLay2026, not by automation.**

### Why not automated

The original workflow at `.github/workflows/upstream-pr.yml` was designed to auto-open cross-repo PRs on merge using a fine-grained PAT (`UPSTREAM_PR_TOKEN`). That design is currently blocked:

- Fine-grained PATs targeting `Bitf1ip/ha-hydros` require Bitf1ip to opt in to fine-grained PAT access for their account. They haven't.
- The classic-PAT alternative (`public_repo` scope) grants write to every public repo the operator collaborates on — blast radius higher than the value of automating one PR per merge.
- Auto-opened cross-repo PRs would also inherit the diff-pollution problem documented below (fork divergence shows up in the upstream diff), so the maintainer would still need a manual triage step.

The workflow file is left in place in a no-op stand-by mode so it can be re-enabled cleanly if either (a) Bitf1ip opts in to fine-grained PATs, or (b) the operator decides a classic PAT's risk surface is acceptable. The workflow header documents the revert procedure.

### Authorship rule

Upstream PRs are opened by **@JLay2026** (the human), not by `github-actions[bot]` or `digispark-bot`. Contributor identity matters — upstream maintainers track contributor history, and service-account PRs dilute that.

Use the GitHub web UI (signed in as @JLay2026) or `gh` CLI authenticated as JLay2026 (`gh auth login`).

### Diff-pollution caveat

`upstream/*` branches are created off `JLay2026/main`, which has no common ancestor with `Bitf1ip/main` (see [#10](../../issues/10)). The cross-repo diff therefore includes fork-only divergence (`manifest.json` customizations, `CHANGELOG.md` differences, fork-specific README sections).

Two handling options:

1. **Recommended:** open the PR with an "Intended files" section in the body listing the files actually meant for upstream review. The maintainer cherry-picks.
2. **Cleaner but more work:** before opening, create a brand-new branch off `Bitf1ip/main`, copy ONLY the intended files onto it, push, then PR from there.

Default to option 1 unless the maintainer signals they'll actually merge.

### Step-by-step

**Prerequisites:** `gh` CLI authenticated as `@JLay2026` (or web-UI access).

1. After the fork PR merges, confirm the `upstream/*` branch still exists on the fork (auto-delete-on-merge is OFF for this repo by default):

   ```
   https://github.com/JLay2026/ha-hydros/branches
   ```

2. **Eyeball the diff first** — open the cross-repo compare URL in a browser:

   ```
   https://github.com/Bitf1ip/ha-hydros/compare/main...JLay2026:ha-hydros:<BRANCH>
   ```

   Confirm the "Files changed" count isn't worse than expected.

3. **Open the PR** via web UI (green button on the compare page) or CLI:

   ```bash
   gh pr create \
     --repo Bitf1ip/ha-hydros \
     --base main \
     --head JLay2026:<BRANCH> \
     --title "<title>" \
     --body-file <path/to/body.md>
   ```

4. **Cross-link the fork PR** so future-you can trace provenance:

   ```bash
   gh pr comment JLay2026/ha-hydros#<FORK_PR> \
     --body "🔗 Upstream PR: https://github.com/Bitf1ip/ha-hydros/pull/<UPSTREAM_PR>"
   ```

5. **Watch and iterate.** Bitf1ip's review cadence is slow — don't chase. If a PR sits 60+ days without engagement, mark it stale locally and move on. The fork is the source of truth.

### PR body template structure

Keep the body focused on:
- **What changed** — one-paragraph summary, then a bulleted list of intended files
- **Why** — link to the fork issue for full context (e.g., `Closes JLay2026/ha-hydros#5`)
- **Test plan** — the `python -m unittest tests.<...>` command, count of passing tests
- **Provenance** — link back to the fork PR; note how long the change has been running in production fork
- **Diff scope** — the "Intended files" list, with an offer to recreate as a clean branch if the maintainer wants to merge

The operator-local file `outputs/upstream-pr-templates.md` contains worked templates for the v0.4.0 hardening sprint PRs as reference examples. (Operator-local because the templates are one-shot artifacts tied to specific already-merged PRs; not committed to avoid stale-doc drift.)

## Daily upstream sync

`.github/workflows/upstream-sync.yml` runs daily at 06:00 UTC (and on-demand via Actions tab → Run workflow).

Behavior:

- **No divergence:** fast-forwards fork's `main` to upstream `main`. No PR, no human review needed.
- **Conflict detected:** opens a `sync/<YYYY-MM-DD>` PR for manual review. Resolve conflicts on that branch, merge to bring fork `main` back in sync.
- **Fork commits ahead:** fork commits are preserved; sync only adds upstream commits. If a fork commit is later merged upstream, the next sync sees it as already-present and skips.

If the daily cadence is too noisy or too sparse, edit the cron expression in the workflow.

**Note:** the sync workflow currently can't fast-forward because fork's `main` has no common ancestor with `upstream/main` (see [#10](../../issues/10)). Until that's resolved, sync behavior is degraded — manual cherry-picks are how upstream changes land on the fork.

## Workflow for hardening items

For each issue in [the M2 tracker (#2)](../../issues/2):

1. **Branch:** `git checkout -b upstream/v0.4.0-<short-name>` (or `fork-only/...`)
2. **Implement:** code + tests + docs against the acceptance criteria in the issue
3. **PR to fork main:** open from the branch; reference the issue number; request your own review or self-merge after CI passes
4. **Merge to fork main:** `Squash and merge` keeps history clean
5. **Manual upstream-PR step** (for `upstream/*` branches only):
   - Open a cross-repo PR against `Bitf1ip/ha-hydros` per "Manual upstream PR submission" above
   - Cross-link the upstream PR URL back to the fork PR as a comment
   - For `fork-only/*` branches: skip this step; merge to fork main is the end
6. **Upstream PR review:** track the upstream PR. Comment, iterate, and merge per Bitf1ip's review cadence. Be patient — the slow upstream velocity is the reason this is a permanent fork.
7. **Upstream merge:** when Bitf1ip eventually merges, the change is already in fork main from step 4 — no follow-up sync needed.
